## 今日の結論
- Phase 1(MVP)の範囲は「既存データ+ルールベースのみ」で実装し、AI API・書き込み可能なバックエンドは導入しない
- ホーム画面の情報順は「企画会議サマリー→今日のおすすめ→急上昇テーマ→候補TOP5→KPI→フィルター→グラフ→動画一覧」に確定
- 公開ダッシュボード(外部クライアント向け・読み取り専用)と、将来の企画ボード(社内向け・書き込み必要)は画面を分離する方針
- レッスン企画・動画企画のタイトル/構成は、Teams投稿と同じ「タグ→テンプレート」のルールベース方式をWeb側にも展開する
- trend_scoreの色分けは絶対値ではなく「表示中データ内の相対順位(四分位)」を採用(動画によって再生数の桁が大きく異なるため)
- 07/20のTeams投稿遅延は、cron-job.orgのトリガー自体は9:00:31に正確に届いており、GitHub Actions側のランナー割り当て待ち(約2時間13分)が原因と判明。コード・cron設定の問題ではない

## 判断理由
- AI(Claude/OpenAI等)によるレッスン企画・動画企画の自動生成は、APIキー取得・課金設定・週次バッチへの呼び出し追加が必要なため、今回はスコープ外とした。既存の`buildReport.ts`のIDEA_TEMPLATES(タグ別テンプレート)と同じ考え方をWeb側に複製することで、AIなしでも「そのまま使える形」を実現した
- 「予想再生数」「難易度」「対応バージョン(統合版/Java)」など、実データで裏付けできない項目は表示に含めなかった。正直に出せる指標(週間再生増加の実績値、投稿数、対象学年・所要時間などテンプレートで確定している内容)のみを採用理由・フォーマットに使用
- 企画ボード(Kanban)やオンデマンドAI生成は、書き込み権限・認証の設計が未確定のため後回しにした。読み取り専用の現行RLS構成を壊さないことを優先
- 複数回の第三者レビュー(85点→92点相当の評価)を踏まえ、各回で「分析結果を企画にどう繋げるか」という同じ弱点を段階的に解消する形で優先順位を決定した

## AIで実施したこと
- 調査: GitHub Actions APIで週次実行の実際のトリガー時刻・ジョブ開始時刻・完了時刻を取得し、Teams投稿遅延の原因(トリガー起因ではなくランナー待ち)を特定
- 比較・検討: 「分析ツール」から「企画意思決定ツール」への転換について、画面構成・データモデル・AI/通常処理の責務分担・MVP〜将来拡張のロードマップを設計し、提案書としてまとめた
- 比較・検討: 複数回の外部レビュー内容を分析し、「既に実装済みの項目」と「本当に未実装の項目」を切り分け(直近レビューでは5項目中4項目が実装済みと判明、キャッシュ起因の誤評価であることを指摘)
- 得られた知見: trend_scoreのような桁の変動が大きい指標は絶対閾値でなく相対順位で色分けするのが適切/GitHub PagesのAI無し・ルールベーステンプレートでも「意思決定支援」に近い体験は作れる/公開データ(読み取り専用)と内部業務データ(書き込み必要)は早い段階で画面を分離しておくと後工程の設計変更が少ない
